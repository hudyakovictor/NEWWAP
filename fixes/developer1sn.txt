# Итерация 1 — Отчёт по анализу кода NEWWAP

Ниже представлены результаты первой итерации: 50 ключевых наблюдений по структуре, логике и потенциальным ошибкам. Анализ основан на полном чтении `run.py` (76 KB), всех модулей `backend/pipeline/` и `backend/core/`.

***

## 1. Архитектура и структура `run.py`

**[A-01]** `run.py` реализует три режима: `calibration`, `main`, `report` — но первые два режима выполняют **идентичный** вызов `extract_one()`. Разница только в источнике `calib_df`: в `calibration` он `None`, в `main` — загружается из CSV. Это создаёт дублирование кода и риск рассинхронизации логики.

**[A-02]** Режим `report` предполагает, что данные уже извлечены (`result.json` существует). Однако нет проверки целостности схемы `result.json` — если схема не совпадает (например, старые файлы без нового поля), `report` будет молча использовать `None`/`0.0` вместо корректных значений.

**[A-03]** Функция `extract_one()` — монолитная: ~670 строк кода в одной функции. В ней смешаны: загрузка изображения, реконструкция, UV-генерация, геометрия, текстурный анализ, калибровка, сохранение. Любая ошибка на промежуточном этапе приводит к потере всех данных для фото.

**[A-04]** Механизм `SKIP (already processed)` проверяет только `status == "ready"`. Если файл записан не до конца (crash в процессе), поле `status` может отсутствовать или быть `"processing"` — тогда файл будет пропущен навсегда без повторной обработки.

**[A-05]** В `run.py` хардкодированы пути `/Users/victorkhudyakov/dutin/...` в `sys.path.insert`. Это критично: если код запускается на другой машине или в Docker, импорты упадут с `ModuleNotFoundError`.

## 2. Ошибки классификации позы

**[P-01]** В `extract_one()` логика `pose_class` (строки ~332–350) содержит **логический баг**: проверка `if ayaw < 15.0: pose_class = "frontal"` выполняется первой, но затем идут проверки `elif yaw < -70.0` и `elif yaw > 70.0` — они никогда не сработают для `ayaw < 15`, всё корректно. Однако ветка `else: pose_class = "unclassified"` достигается только если `10 < ayaw < 15` но `yaw > 0` — то есть лёгкий правый разворот в диапазоне 10–15° попадает в `unclassified` вместо `frontal` или `right_threequarter_light`.

**[P-02]** В `bucket_from_filename()` (строки 17–32) используется **другая шкала** порогов: `< 15` → frontal, `< 25` → threequarter_light, `< 45` → threequarter_mid, `< 70` → threequarter_deep. В `extract_one()` пороги: `< 15` frontal, `10–25` light, `25–45` mid, `45–70` deep. Несоответствие: в `bucket_from_filename` нет порога 10°, а диапазон 10–15° относится к frontal через общий `ayaw < 15`. Это ведёт к тому, что имя файла и вычисленный bucket могут не совпадать.

**[P-03]** В `reconstruction.py` (строки ~204–222) **дублируется** та же логика классификации позы — третья копия. Все три копии имеют слегка разные пороги. Нужен единый `classify_pose(yaw)` в `types.py` или `constants.py`.

**[P-04]** `angles_deg` из 3DDFA-V3 возвращает `[pitch, yaw, roll]`. В `pose_result` корректно: `yaw = res.angles_deg[1]`. Но в нескольких местах `run.py` встречается `_raw_angles = np.array([pitch, yaw, roll])` — порядок компонент соответствует документации 3DDFA-V3, это правильно.

**[P-05]** Нет проверки на `NaN` в `angles_deg` после реконструкции. Если 3DDFA-V3 вернёт вырожденную позу, все последующие вычисления пройдут с `NaN`-значениями, которые молча распространятся в JSON.

## 3. Маппинг вершин и трансформации

**[V-01]** Инверсия Y-координаты вершин изображения: `v2d_img[:, 1] = 224.0 - 1.0 - v2d_img[:, 1]`. Это стандартная операция для перехода из OpenGL (Y вверх) в image space (Y вниз). Однако константа `224.0` хардкодирована — размер входного тензора 3DDFA-V3. Если когда-либо изменить входной размер (например до 256), это сломается без предупреждения.

**[V-02]** Back-mapping из 224×224 в оригинальный размер изображения:
```python
v2d_mapped[:, 0] = (v2d_mapped[:, 0] + left) / w * w0
v2d_mapped[:, 1] = (v2d_mapped[:, 1] + up) / h * h0
```
Формула математически корректна для аффинного обратного преобразования при условии, что `trans_params` содержит `[w0, h0, s, t0, t1]` именно в таком порядке. Необходимо верифицировать формат `trans_params` из 3DDFA-V3 документации — в разных версиях библиотеки порядок мог меняться.

**[V-03]** Ветка `else: v2d_mapped = v2d_img` когда `trans_params is None` означает, что координаты остаются в пространстве 224×224 вместо оригинального изображения. Это приведёт к некорректной проекции UV-текстуры при генерации — без ошибки, но с визуально неверным результатом.

**[V-04]** `verts_3d[:, 2] = -verts_3d[:, 2]` — инверсия Z для UV-генератора. Комментарий `# Fix Z-buffer occlusion` не объясняет почему нужна инверсия. При этом канонизированные вершины уже прошли через `canonicalize_vertices_for_bucket()` — нет гарантии что двойная трансформация (canonicalize + Z-flip) не вносит артефактов для профильных фото.

**[V-05]** `normals_camera` используется для расчёта confidence mask UV: `confidence_per_vertex = np.power(np.clip(res.normals_camera[:, 2], 0, 1), 4.0) * 255`. Степень `4.0` — очень агрессивное подавление. Нормаль с `z=0.7` (разворот ~45°) даст confidence `0.7^4 = 0.24` — т.е. 75% подавления для полуфронтальных нормалей. Это излишне строго и может занижать уверенность для рабочих зон лица.

## 4. UV-текстура и маска

**[U-01]** UV-генерация вызывается с `vertices_canon_for_uv` (канонизированными), но `v2d_mapped` — это 2D-проекция **оригинальных** вершин (до канонизации). Передача канонизированных 3D-вершин с некаонизированными 2D-проекциями в `HDUVTextureGenerator` логически несогласована и может давать неверный barycentric bake.

**[U-02]** После UV-генерации создаётся **второй** fallback mask из `normals_camera`. Но итоговая маска (`uv_confidence_mask`) затирает маску из `HDUVTextureGenerator` (`uv_mask_visible`). Зачем тогда первая маска из генератора? Она нигде не сохраняется. Дублирование с потерей данных.

**[U-03]** Морфологическое эродирование маски: `cv2.erode(uv_confidence_mask, kernel, iterations=1)` с `kernel = np.ones((5,5))`. Это убирает 2–3 пикселя с краёв, что при UV-размере 1024 px соответствует ≈0.2–0.3% площади. Эффект минимальный, но комментарий `# [FIX TX-03]` предполагает что это решает проблему чёрных пикселей фона. Реальное решение — улучшить маску сегментации, а не эрозию конечного результата.

**[U-04]** `uv_tex_analysis` (результат анализа от HDUVTextureGenerator) записывается в `texture_preds`. Позже при калибровке из `texture_preds` берётся `quality_quality_index` через цепочку `.get()` с fallback `0.7`. Если HDUVTextureGenerator вернёт другой ключ — тихо используется дефолтное `0.7`, что может систематически занижать/завышать калибровку.

**[U-05]** `fillConvexPoly` для UV-маски — ошибочный подход: треугольники лица не являются строго выпуклыми. Правильная функция — `fillPoly` или растеризация через `drawContours`. При сильных деформациях меша `fillConvexPoly` будет заполнять неверную область.

## 5. Сегментационная маска кожи

**[S-01]** Каналы сегментации 3DDFA-V3: в коде закомментировано `# [right_eye, left_eye, right_eyebrow, left_eyebrow, nose, up_lip, down_lip, skin]`. По документации 3DDFA-V3 (`seg_visible` из `raw_result`) канал **8** — фон, а не skin. Нужна верификация: skin = канал 7 или 8.

**[S-02]** Исключение каналов: `for i in [0, 1, 2, 3, 5, 6]` убирает глаза, брови, губы. Канал `4` (нос) — **не исключается** из маски кожи. Это значит текстурный анализ включает зону носа, но геометрические метрики носа могут измеряться отдельно — возможен double-counting в некоторых метриках.

**[S-03]** `back_resize_crop_img` из `util.io` (3DDFA-V3 utility) принимает изображение `uint8`. Передаётся `np.uint8) * 255` — бинарная маска умноженная на 255. Это корректно для uint8. Однако если `skin_224` содержит float-значения (soft segmentation), умножение и приведение к uint8 даст бинаризацию по порогу 0, теряя градиентную уверенность сегментации.

***

## 6. Калибровка и `_compute_pair_calibration`

**[C-01]** `find_calibration_match` вызывается с `bucket=pose_class`, но в `calib_df` колонка переименована с `pose_bucket` → `bucket` только для этого вызова (`_calib_for_match`). Однако в дальнейшем вычислении `geo_prefix_cols = [c for c in calib_df.columns...]` используется **оригинальный** `calib_df` (без переименования). Если код пытается обратиться к `bucket` в оригинальном df — ошибки KeyError. Фактически код не использует оригинальный df для этой цели, но неконсистентность опасна.

**[C-02]** `pair_noise` строится через `combinations(top_k_records, 2)` — межпарные абсолютные дельты между калибровочными снимками. При `k=5` это 10 пар. Это разумная оценка шума позы. Однако `pair_noise[metric]` = `mean(abs_deltas)` без нормализации на масштаб метрики. Для метрик разных диапазонов (например, угол в градусах vs. ratio 0–1) noise baseline несопоставим.

**[C-03]** `GEO_EXCLUSION_WHEN_JAW` и `GEO_EXCLUSION_WHEN_SMILE` — правильная идея исключения нестабильных метрик. Но пороги для определения `jaw_excluded` и `smile_excluded` нигде не видны в этом фрагменте. Если они установлены слишком жёстко — многие фото будут исключать критические метрики (гониальный угол и т.д.), занижая покрытие анализа.

**[C-04]** Fallback при пустом `calib_df`: возвращается `global_noise_baseline: 0.015`. Это хардкодированная константа без обоснования. Для разных метрик (ratios 0.01–2.0, углы 0–90°) одно значение 0.015 не может быть универсальным noise baseline.

**[C-05]** `_compute_linear_snr` импортируется но нигде в этом фрагменте не используется напрямую — только `find_calibration_match`. Возможно, планировалось использовать SNR для взвешивания, но не реализовано.

## 7. Геометрические метрики (`scoring.py`, `zones.py`)

**[G-01]** `extract_macro_bone_metrics` содержит функцию `get_zone_centroid(name)` — берёт среднее по точкам зоны. Центроид зоны лица — не то же самое, что анатомическая landmark-точка. Для метрик типа `gonial_angle` нужна конкретная точка мыщелка, а не центроид зоны челюсти.

**[G-02]** `calc_tilt_3d_coronal` вычисляет наклон через проекцию на корональную плоскость с использованием `face_normal`. При профильных ракурсах `face_normal` может быть нестабильным — это ведёт к неверным tilt-метрикам именно там, где они наименее надёжны.

**[G-03]** `_robust_trimmed_3d_error` с `[ITER-1.2]` тримирует выбросы по квантилю 0.15 (15% с каждой стороны). Для малых наборов вершин зоны (например, 20–30 точек) тримирование убирает 3–4 точки — возможно полезные данные, а не выбросы.

**[G-04]** `score_aligned_pair` использует `effective_weights = weights * reliability_weight` — скалярный множитель надёжности. Если `reliability_weight = 0.3` (плохое фото), все веса уменьшаются в 3.3×, но **относительные** веса между зонами не меняются. Фактически reliability_weight не влияет на итоговый score через `weighted_mean_abs` (числитель и знаменатель масштабируются одинаково). Это **ошибка**: reliability должен влиять на доверительный интервал, а не на сами веса при нормировке.

**[G-05]** `fit_best_plane` для face plane через SVD — стандартный подход. Однако нет проверки на degenerate case: если все точки коллинеарны (artifical/synthetic face или сильно профильный снимок), SVD вернёт вырожденную нормаль.

***

## 8. Продольный анализ (`longitudinal.py`)

**[L-01]** `build_longitudinal_model` принимает `summaries` — список dict с `parsed_date`. Дата парсится как строка `"YYYY-MM-DD"`. Сортировка по строке работает корректно только если формат строго ISO. Если хотя бы одна дата имеет формат `"DD.MM.YYYY"` — сортировка будет неверной.

**[L-02]** Аномалии сериализуются как `anom['metric']`, но в коде `anomalies_serialized` используется `anom.metric_key` через `getattr`. Внутри `suspicious_windows` обращение — `anom['metric']` (dict-style). Несоответствие стилей доступа к атрибутам объекта vs. dict.

**[L-03]** `aging_consistency_score` из `get_summary()` — единственный агрегатный показатель долгосрочной стабильности. Его формула не видна в этом фрагменте. Если он просто инвертирует количество аномалий — это упрощение, не учитывающее severity и временные паттерны.

***

## 9. Вердикт и глобальный статус

**[VD-01]** `global_status` меняется на `"unresolved_anomalies"` при первом же `"different_person"` в любом bucket. Нет механизма "веса bucket-а": single anomaly в ненадёжном `profile` bucket имеет такой же вес, как anomaly во `frontal`. Это ведёт к ложным тревогам.

**[VD-02]** `confidence` вычисляется как weighted average H0 по всем парам. Но `all_reliability` берётся только для `photo_a` каждой пары. `photo_b` reliability игнорируется. Геометрическое среднее reliability обоих фото было бы точнее.

**[VD-03]** `suspicious_windows` заполняется из `anom['metric']`, но в `anomalies_serialized` ключ называется `metric_key`. При переборе это даст `KeyError` или пустую строку в зависимости от реализации `Anomaly` объекта.

**[VD-04]** Итоговый `summary` текст в `global_verdict` — хардкодированная строка на русском. При любом статусе кроме `same_person` пишется стандартный текст без конкретики. Никакой динамической генерации объяснения на основе реальных флагов.

***

## 10. Модули `compare.py`, `verdict.py`

**[CV-01]** В `verdict.py` — логика перевода метрик в вердикт H0/H1/H2. Без детального чтения на этой итерации: критично проверить что пороги вердикта согласованы с noise baseline из калибровки. Несогласованность пороги/шум — главный источник ложных срабатываний.

**[CV-02]** `compare.py` выполняет попарное сравнение фото. Порядок пар в `timeline_by_bucket` зависит от сортировки по дате. Если дата не распознана (`parsed_date = None`), фото с `None`-датой попадают на позицию `"1900-01-01"` — в начало хронологии. Это искусственно "растягивает" временной ряд.

## Главные выводы первой итерации

Три **критических** проблемы, способных давать неверные выводы:

**1. Тройное дублирование логики классификации позы с расходящимися порогами** (P-01, P-02, P-03) — разные модули могут присваивать одному фото разные bucket-ы, что ломает калибровку.

**2. Несогласованность 3D/2D пространств в UV-генерации** (U-01) — канонизированные 3D-вершины + некаонизированные 2D-проекции = некорректный barycentric bake = неверный текстурный анализ.

**3. Reliability weight не влияет на score** (G-04) — математический баг: скалярный множитель на все веса аннулируется при нормировке в `weighted_mean_abs`, делая механизм надёжности декоративным.

# Итерация 2 — Углублённый анализ `zones.py`, `verdict.py`, `alignment.py`, `compare.py`, `calibration.py`, `visibility.py`

***

## 1. `zones.py` — Индексы зон и геометрия

**[Z-01]** `EAR_INDICES = {13, 14, 15, 16}` — четыре индекса. В BFM/3DDFA-V3 меш содержит ~53000 вершин, из которых ушные зоны занимают сотни индексов. Четыре хардкодированных индекса не могут адекватно закрыть всю ушную область — `filter_indices` фактически не выполняет своей функции. Любая зона, включающая индексы ушей вне {13,14,15,16}, будет загрязнять метрики ушными артефактами.

**[Z-02]** `cheekbone_R` присутствует в `ZONE_CONFIG` и `MACRO_BONE_INDICES` с полноценным набором индексов (~200+ вершин). Но в `extract_macro_bone_metrics` в `scoring.py` написан fallback: `if np.allclose(cheek_R, 0): ... cheek_R = cheek_L`. Это значит что при любом сбое в `get_zone_centroid('cheekbone_R')` правая скула будет зеркалом левой, что даст симметричный `zygomatic_breadth = 0` — нулевой знаменатель для всех ratio-метрик. Вместо fallback нужна явная валидация индексов.

**[Z-03]** `canthus_L_inner`, `canthus_L_outer`, `canthus_R_inner`, `canthus_R_outer` — в `scoring.py` пытается взять их через `_idx('canthus_L_inner')`, но эти ключи **отсутствуют** в `MACRO_BONE_INDICES` (словарь содержит только `orbit_L`, `orbit_R`, `jaw_angle_L` и т.д.). Код всегда попадает в ветку fallback: `canthus_L_inner = orbit_L_pts[np.argmin(orbit_L_pts[:, 0])]`. Кантальный наклон (`canthal_tilt_3d_L/R`) всегда вычисляется по краям bounding box орбиты, а не по реальным кантальным точкам. Это систематическая ошибка — метрика `canthal_tilt` заявлена как точная, но измеряется грубым proxy.

**[Z-04]** В `ZONE_CONFIG` указаны зоны `ligament_zygomatic_L/R`, `ligament_orbital_L/R`, `temporal_L/R`, `cheek_soft_L/R` — но ни одна из них не присутствует в `MACRO_BONE_INDICES`. Если какой-то код обращается к этим зонам через `_idx()`, всегда получает пустой массив. Вес `bone_priority_core` для `jaw_angle_L/R` (1.0) тоже задан в ZONE_CONFIG, но индексы в MACRO_BONE_INDICES — не проверить без полного парсинга файла 99KB.

**[Z-05]** `EXCLUDED_ZONES` содержит `cheek_lower_L/R` — но в `ZONE_CONFIG` эти зоны называются `cheek_soft_L/R`. Несоответствие имён означает, что `cheek_lower_L/R` никогда не присутствовали в pipeline (нечего исключать), а `cheek_soft_L/R` не исключаются там, где должны бы. Это тихая рассинхронизация имён.

**[Z-06]** `apply_expression_exclusion_mask` использует `exp_params[0]` как jaw-open и `max(abs(exp_params[1]), abs(exp_params[2]))` как smile. По документации 3DDFA-V3 expression basis использует FLAME expression coefficients: первые 3 компоненты (`exp_params[0..2]`) отвечают за глобальные деформации, из которых `exp_params[0]` — первая PCA-компонента (обычно открытие рта), `exp_params[1..2]` — вторая и третья (могут быть smile, но строгого маппинга по документации 3DDFA-V3 нет). Нет ссылки на исходный код 3DDFA-V3 с подтверждением этих индексов.

**[Z-07]** `THRESHOLD_JAW_OPEN = 0.22` и `THRESHOLD_SMILE = 2.2` — определены **дважды**: в модуле `zones.py` (строка ~30) и внутри функции `apply_expression_exclusion_mask` (строка ~50). Локальные переменные функции затеняют модульные константы. Если изменить модульный порог — функция не изменит поведения. Нужна единственная точка определения через `constants.py`.

## 2. `alignment.py` — Канонизация и Umeyama

**[AL-01]** `canonicalize_vertices_for_bucket` использует `Rotation.from_euler('xyz', [-pitch, -yaw, -roll])`. Знак минус применяется ко всем трём осям одновременно. Инверсия знаков для конвертации из системы 3DDFA в стандартную СК должна быть обоснована для каждой оси отдельно. В 3DDFA-V3 pitch — это вращение вокруг X, yaw — вокруг Y, roll — вокруг Z, при этом знак yaw уже учитывает right-hand convention. Инверсия `-yaw` может приводить к неверному направлению поворота для правосторонних разворотов.

**[AL-02]** В `align_canonical_pair_for_view_group` (строка ~107–148): `avg_pitch = (angles_a[0] + angles_b[0]) / 2.0` — арифметическое среднее углов используется как target. Но если `photo_a` имеет pitch=+15° (голова запрокинута), а `photo_b` имеет pitch=-15° (голова наклонена), target=0° — это не описывает ни одно из фото. Для двух снимков с существенно разным pitch геометрическое сравнение в такой "усредненной" плоскости будет вносить систематическую ошибку.

**[AL-03]** `rigid_umeyama` с `allow_scale=False` используется для alignment при вычислении score, но в `align_canonical_pair_for_view_group` — `allow_scale=True` (строка 148). Это значит, что при парном сравнении через `PairComparisonEngine` применяется GPA unit-scale нормировка + затем ещё раз масштабирование Umeyama. Двойное масштабирование может занижать реальную ошибку сравнения.

**[AL-04]** `gpa_unit_scale` использует `sqrt(mean(sum(p_i^2)))` — это RMS distance от центроида, стандартная Procrustes нормировка. Однако в `PairComparisonEngine` выбор между `scale_a_gpa` и `scale_a_ipd` (межзрачковое расстояние) происходит по условию наличия `orbit_L/R` в `bone_indices`. При отсутствии орбитальных зон (профильные снимки) используется GPA-scale — что корректно. Но при наличии орбитальных зон IPD-scale в абсолютных единицах 3DDFA (мм-подобных) может значительно отличаться от GPA-scale: это ломает сопоставимость score между парами, нормированными разными способами.

**[AL-05]** В `euler_to_rotation_matrix` (строки ~89–105): реализация через ручное перемножение матриц `Rz @ Ry @ Rx`. Порядок вращений `ZYX` — это Tait-Bryan углы в порядке yaw-pitch-roll. Но 3DDFA-V3 возвращает углы в BFM-convention (XYZ). Использование ZYX-порядка для XYZ-углов даст неверную матрицу поворота в `align_canonical_pair_for_view_group` (эта функция использует `euler_to_rotation_matrix`, а не `scipy.Rotation`).

## 3. `visibility.py` — Маска видимости

**[VS-01]** `compute_software_zbuffer` использует 256×256 сетку для Z-buffer. При меше ~53000 вершин лица это ~0.8 вершины на ячейку в среднем, но в центральных зонах (нос, орбиты) плотность намного выше. Разрешение 256×256 **слишком грубое** для детекции самоокклюзии в мелких структурах — нос будет частично "просвечивать" сквозь себя при профильных снимках.

**[VS-02]** `np.minimum.at(z_buffer, (y_idx, x_idx), z)` — записывает минимальный Z в каждую ячейку. Но при множественных вершинах в одной ячейке все они сравниваются с этим минимумом. Вершина с `z = z_min + epsilon` будет считаться видимой, хотя геометрически она за другой вершиной. `epsilon = (z_max - z_min) * 0.005` = 0.5% от всего диапазона глубины — при глубоком профиле это может быть 5–10 мм, что недостаточно мало.

**[VS-03]** `get_visible_zones(yaw, pitch)` — функция использует строки-имена зон (`"left_eye"`, `"right_zygomatic"`) которые НЕ соответствуют именам в `ZONE_CONFIG` (`"left_eye"` → `"orbit_L"`, `"right_zygomatic"` → `"cheekbone_R"`). Функция никогда не вернёт корректный список зон, совместимый с остальным pipeline. Если она где-то используется для фильтрации — результат всегда будет некорректным.

**[VS-04]** Затухание весов при `45° ≤ yaw ≤ 60°`: `turning_away = x_coords < 0` (для `yaw > 0`). Проверка по знаку X в camera-space — это грубое приближение. Вершины правой скулы при yaw=+50° имеют X<0, но они всё ещё частично видимы. Корректнее использовать нормаль вершины, как уже делает `facing_cosines`. Двойная проверка (нормаль + X-знак) излишне жёсткая.

## 4. `compare.py` — Логика попарного сравнения

**[CP-01]** В `PairComparisonEngine.compare()` оба фото канонизируются к `view_group_a`: `canonicalize_vertices_for_bucket(verts_b, angles_b, view_group_a)`. То есть если `photo_a` — frontal, а `photo_b` — threequarter_mid, то `photo_b` будет канонизировано к frontal target_yaw=0°. Это огромное принудительное вращение (~30°) вносит геометрические артефакты в меш `photo_b`. Правильный подход: использовать `view_group` того из двух фото, что ближе к фронтальному.

**[CP-02]** `shared_vertex_indices(vis_a, vis_b)` берёт пересечение видимых вершин обоих фото. При сравнении frontal и profile пересечение будет минимальным (~20–30% вершин). Порог `ALIGNMENT_MIN_RANK = 3` — катастрофически низкий: три вершины дают вырожденный Umeyama с нестабильной матрицей поворота. Нужен порог не менее 50–100 вершин.

**[CP-03]** `matches_b = find_calibration_match(..., bucket=view_group_a, k=5)` — поиск калибровочных пар для `photo_b` по bucket `photo_a`. Это неверно: если `photo_b` имеет другой bucket, то ближайшие калибровочные фото для него должны искаться по его собственному bucket.

**[CP-04]** `local_noise[zone] = float(local_pairs[col].median())` — медиана попарных ошибок из калибровочных пар используется как noise baseline. Медиана ошибок же-человека корректна как оценка шума. Но `col` строится как `f"zone_error_{zone}"` — нужно проверить что `calib_pairs_df` действительно содержит такие колонки; если нет — `local_noise` будет пустым и вся local calibration silent-упадёт в fallback.

**[CP-05]** После GPA нормировки: `points_a_unit = (verts_a_canon - centroid_a) / (scale_a + 1e-8)`. `centroid_a` вычислен по `shared_idx` вершинам, а вычитается из **всех** `verts_a_canon`. Это правильно — центроид видимой части использован как reference, и нормировка применяется ко всему мешу. Но затем `points_a_unit[shared_idx]` — снова берутся shared вершины. Полный меш нормируется без нужды. Незначительная неэффективность, не ошибка.

## 5. `verdict.py` — Байесовский движок

**[VD2-01]** `PRIOR_SAME_PERSON = 0.65`, `PRIOR_IDENTITY_SWAP = 0.02` из `constants.py`. `H2_prior = 1 - 0.65 - 0.02 = 0.33`. При апостериорном обновлении логарифмами: `_update_posteriors_log_domain`. Однако лайклихуды геометрии и текстуры передаются как **абсолютные значения** (не нормированные). Если `l_geom_h0 = 0.95` и `l_geom_h2 = 0.90`, отношение лайклихудов `0.95/0.90 = 1.055` — очень слабый сигнал. При таком различии Байес не сдвинет posterior значимо, что корректно. Но если оба равны `0.5`, posterior равен prior — это тоже правильно. Проблема в том, что нет описания того, откуда берутся конкретные значения `l_geom_h0` / `l_geom_h1` — они зависят от SNR-порогов, которые **не калибруются** из данных, а хардкодированы.

**[VD2-02]** `confidence = _clamp01(p_h0 * (1.0 - tex * 0.5))` для `SAME_PERSON` статуса (строка ~270). При `tex = 1.0` (максимальный силиконовый сигнал) confidence = `p_h0 * 0.5`. Это интуитивно понятная аттенюация, но формула линейная — при `tex = 0.3` уменьшение всего 15%. Пороговая чувствительность к силикону слабая.

**[VD2-03]** `is_impossible = any(f["type"] == CHRONO_FLAG_IMPOSSIBLE ...)` + `status = ForensicStatus.IDENTITY_SWAP`. Единственного хронологического флага "impossible" достаточно для вынесения вердикта IDENTITY_SWAP. Хронологические флаги генерируются в `chronology.py` по временным правилам. Если эти правила дают ложный флаг — вердикт автоматически становится неверным без возможности override на основе сильной геометрической H0-уверенности.

**[VD2-04]** `confidence_cap = 0.75` для `FALLBACK` режима геометрии. Но `SAME_PERSON` при FALLBACK + хорошем геометрическом score может дать только 0.75 уверенности, тогда как `IDENTITY_SWAP` при хронологическом флаге — `max(0.90, impossible_prior)`. Асимметрия: "тревога" имеет выше возможный confidence, чем "норма". Это может быть намеренным, но требует обоснования в ТЗ.

## 6. `calibration.py` — Noise Model

**[CAL-01]** `NoiseModel._rebuild()` вычисляет `errors = [obs.zone_errors[zone] for obs in self.observations if zone in obs.zone_errors]`. Затем `base_mad = np.median(errors)`. Медиана **абсолютных** ошибок — это корректная оценка MAD при условии, что ошибки симметрично распределены вокруг нуля (same-person comparison). Однако `obs.zone_errors` хранит **не signed errors**, а что именно — зависит от того, как строится `NoiseObservation`. Если там хранятся `raw_error` (всегда ≥0), то `median(raw_errors)` = MAD по определению. Корректно.

**[CAL-02]** `ZoneNoiseProfile.predict_noise(pose_delta_mag)` = `base_mad + pose_delta_mag * 0.02`. Линейная модель: каждый градус угловой разницы добавляет `0.02` к baseline шуму. При `pose_delta = 30°` добавка `0.6` — больше типичного base_mad (~0.01–0.05). Это означает что при любом существенном угловом несоответствии (>5°) шум модели определяется не калиброванным base_mad, а хардкодированным коэффициентом `0.02`. Этот коэффициент не калибруется из данных.

**[CAL-03]** `decompose_with_local_noise` получает `local_noise: Dict[str, float]` из `compare.py`. Если `local_noise` пустой (ситуация из CP-04) — вся local calibration деградирует в `decompose()` с глобальной моделью. Это silent fallback без предупреждения в логах.

## Почему `EAR_INDICES = {13, 14, 15, 16}` — ошибка

В BFM/3DDFA-V3 меше **уши являются частью поверхности "skin"** (8-я группа аннотации). Отдельной аннотационной группы для ушей нет. Индексы 13, 14, 15, 16 — это просто четыре произвольных вершины где-то в начале меша, никак не связанных с ушными раковинами анатомически.

Реальная ушная область в BFM-топологии занимает несколько **сотен** вершин в диапазоне приблизительно 8000–12000, и они никак не помечены специальным образом — потому что 3DDFA-V3 **не реконструирует уши**. Форма ушей в 3DDFA-V3 является **статической средней формой из BFM** без подгонки под конкретного человека.

## 1. `texture.py` — Критические проблемы метрик

**[TX-01]** `silicone_probability` — **поле-призрак**. В `TextureMetrics` его нет. `SkinTextureAnalyzer` не вычисляет `silicone_probability` нигде. Но `core/analysis.py` извлекает `tex_a.get("silicone_probability")` в `_compute_texture_h1_evidence()` (строки 432–433) и использует как главный вес 0.35 в итоговом H1-score. Возвращается `None` → весь H1-evidence движок работает только на `fft_high_freq_ratio` и `lbp_uniformity`. Аналогично `fft_high_freq_ratio` и `albedo_uniformity` — их нет в `TextureMetrics` и они нигде не вычисляются. Три из пяти ключевых признаков силикона/синтетики отсутствуют в pipeline фактически.

**[TX-02]** `analyze_image()` — это wrapper, вызывающий `analyze()` и конвертирующий результат `TextureMetrics` → `dict` через `asdict()`. Но `run.py` вызывает `analyzer.analyze_image(...)` и ожидает плоский `dict`. После `asdict()` все поля правильно сплющены. **Однако:** `quality_quality_index` — такого ключа нет. В `TextureMetrics` поле называется `quality_index`. При `asdict()` оно станет `"quality_index"`. В `run.py` есть двойной поиск: `texture_preds.get("quality_quality_index") or texture_preds.get("quality_index")` — вторая ветка работает. Но в `_compute_pair_calibration()` (строка 100) используется только первая ветка `quality_quality_index` без fallback → всегда `None` → `quality_weight = 0.7` (хардкодированный дефолт).

**[TX-03]** GLCM вычисляется на **полном изображении** (`gray_32 = (gray // 8).astype(np.uint8)`), а не на замаскированных пикселях. Затем `graycomatrix(gray_32, ...)` включает все пиксели — включая фон, одежду, волосы. Маска `skin_mask` применяется только к LBP и Gabor. Для GLCM-метрик (contrast, energy, homogeneity, correlation) это означает систематическое загрязнение фоновыми текстурами. По документации scikit-image `graycomatrix` принимает 2D-массив целиком — нет параметра маски. Корректное решение: `gray_masked_2d = gray_32.copy(); gray_masked_2d[skin_mask == 0] = 0` — тогда нулевые пиксели станут частью GLCM, но их вклад через `normed=True` будет подавлен.

**[TX-04]** `specular_gloss`:
```python
p97 = np.percentile(masked_pixels, 97)
bright_threshold = min(float(p97), 230.0)
metrics.specular_gloss = float(np.sum(masked_pixels > bright_threshold) / masked_pixels.size)
```
Логика инвертирована. `bright_threshold = min(p97, 230)`: если `p97 < 230` — порог равен `p97`, и всегда ≥3% пикселей будут выше → `specular_gloss ≥ 0.03` всегда. При тёмной коже `p97` может быть 150 → порог 150 → 3%+ пикселей выше 150 всегда. `min()` должен быть `max()`: порог не ниже 230 — тогда только реально яркие блики войдут в счёт.

**[TX-05]** `crow_feet` ROI: `_roi_by_fraction(gray, 0.15, 0.45, 0.0, 0.20)` — левый глаз занимает x: 0–20% ширины crop. Это работает только если `face_crop` — именно кроп лица с центрированием. Если кроп берётся из bounding box 3DDFA (который включает голову от темени до подбородка), гусиные лапки у внешних углов глаз будут в зоне x: 0–10%, а не 0–20%. ROI слишком широкий, захватывает часть носа/лба левой половины лица.

**[TX-06]** `quality_index` formula:
```python
pose_factor = np.cos(np.radians(yaw_deg)) * np.cos(np.radians(pitch_deg))
metrics.quality_index = float(np.clip(q_index_sharp * pose_factor, 0.0, 1.0))
```
При `yaw=45°, pitch=0°`: `pose_factor = cos(45°) = 0.707`. При `quality_sharpness_score > 400` → `q_index_sharp ≈ 0.90` → `quality_index ≈ 0.636`. Это разумно. Но при `pitch=20°`: `pose_factor = cos(45°) * cos(20°) = 0.707 * 0.940 = 0.665`. Итого `quality_index ≈ 0.60` — излишне пессимистично для фото с хорошей резкостью. **Проблема**: `quality_index` используется как мультипликатор к `reliability_weight` в `scoring.py`. Двойное снижение через `pose_factor` в texture и через `if yaw_abs > 30: reliability *= 0.5` в scoring означает что профильные фото получают reliability ≈ `0.5 * 0.637 = 0.32` — почти игнорируются при калибровке.

**[TX-07]** UV-анализ: `uv_gray = cv2.imread(str(uv_path), cv2.IMREAD_GRAYSCALE)` — UV-текстура сохраняется как JPEG (`uv_texture_hd.jpg`). JPEG-артефакты на высококонтрастных границах UV напрямую влияют на `uv_wrinkle_energy = mean(laplace^2)` и `uv_retouch_score = sum(lbp_hist^2)`. Сохранение UV как PNG устранило бы систематический шум от JPEG-компрессии в текстурных метриках.

## 2. `texture.py` — Менее критичные, но важные

**[TX-08]** `laplace()` из scikit-image (`skimage.filters.laplace`) по умолчанию использует `ksize=3` Лапласиан. Но `cv2.Laplacian(gray, cv2.CV_64F)` в `quality_sharpness_score` (строка 257) — другая реализация. Оба называются "Laplacian", но дают разные величины: sklearn-laplace нормирован, cv2-Laplacian — нет. `laplacian_energy` и `quality_sharpness_score` несопоставимы по шкале, хотя оба претендуют на "меру резкости".

**[TX-09]** LBP `bins=10, range=(0, 10)` — при `method="uniform"` LBP с P=8 даёт 10 uniform patterns (0–9). Диапазон `(0, 10)` исключает значение 10 (non-uniform pattern, который кодируется как P+2=10). По документации scikit-image `local_binary_pattern` с `method="uniform"` возвращает значения от 0 до P+2=10 включительно. Значение 10 (non-uniform) — самый информативный бин для обнаружения texture edges — систематически теряется.

**[TX-10]** Gabor в цикле: три частоты [0.1, 0.2, 0.3], угол по умолчанию 0 радиан. Ориентация фиксирована — горизонтальные Gabor-фильтры. Морщины на лбу ориентированы горизонтально — это подходит для `wrinkle_forehead`. Но для `crow_feet` и `nasolabial_depth` морщины диагональные. Single-orientation Gabor даёт систематически заниженный отклик для нехоризонтальных морщин.

## 3. `scoring.py` — Метрики и веса

**[SC-01]** `gonial_angle` fallback при отсутствии `gonion_L`:
```python
metrics["gonial_angle_L"] = angle_R  # Symmetric fallback
metrics["gonial_angle_R"] = angle_L  # Symmetric fallback
```
В блоке "правый gonion найден но левый нет" — `gonial_angle_L = angle_R`. В блоке "левый найден но правый нет" — `gonial_angle_R = angle_L`. И дополнительно `metrics["gonial_angle_R"] = 0.0` при полном отсутствии обоих. `gonial_angle = 0.0` — физически невозможное значение, даже у трупа угол ≥ 100°. При сравнении со здоровым лицом (gonial_angle ≈ 130°) это даст Delta = 130° — ложный сигнал IDENTITY_SWAP.

**[SC-02]** `reliability_weight` в `score_aligned_pair` (итерация 1, баг G-04 подтверждается здесь):
```python
effective_weights = weights * reliability_weight
# затем:
robust_error = _robust_trimmed_3d_error(distances_normalized, effective_weights)
```
`_robust_trimmed_3d_error` использует `effective_weights` как `np.average(trimmed_dists, weights=trimmed_w)`. При `np.average` с весами — результат равен `sum(w_i * d_i) / sum(w_i)`. Масштабирование всех `w_i` на `reliability_weight` отменяется при делении. **Баг подтверждён**: `reliability_weight` не влияет на итоговый `score`.

**[SC-03]** `face_plane_normal` для `calc_tilt_3d_coronal` вычисляется через SVD по ~200 точкам skin-зоны. При `yaw > 45°` половина этих точек находится "за горизонтом" лица и невидима, но всё равно участвует в SVD (они в 3D-меше). Normal плоскости смещается в сторону неправильного направления. `canthal_tilt_3d` для профильных ракурсов будет систематически неверным, даже если бы кантальные индексы были корректными.

## 2. `chronology.py` (pipeline) — Логика временных аномалий

**[CH-01]** `analyze_timeline` сортирует фото внутри bucket по `extracted_at`:
```python
bucket_photos.sort(key=lambda x: x.get("extracted_at", ""))
```
`extracted_at` — timestamp обработки (`datetime.now().isoformat()`), а не дата фотографии. Хронологический анализ строится по **времени обработки**, а не по дате съёмки. Фото, обработанное позже, будет "позже" по timeline — независимо от реальной даты. Нужно сортировать по `parsed_date` из имени файла.

**[CH-02]** RTR-детектор вычисляет `ref_avg`:
```python
r_vec, _ = get_ordered_metric_vectors(p.get("metrics", {}), p.get("metrics", {}), ...)
ref_metrics.append(r_vec)
ref_avg = np.mean(ref_metrics, axis=0)
```
`get_ordered_metric_vectors(A, A, keys)` — оба аргумента одинаковые, функция возвращает `(v1, v2)` где `v1 == v2`. Используется только `r_vec = v1`. Это корректно, но избыточно и запутанно — функция создана для сравнения двух разных векторов. Лучше использовать `get_ordered_metric_vector`.

**[CH-03]** `SuspiciousWindow._calculate_p_substitution`: при `days < IMPOSSIBLE_AGE_REVERSAL_DAYS = 180` → `prior_p_substitution = 0.8`. При `days < 90` → `0.4`. Но порядок условий **инвертирован**:
```python
if self.days < IMPOSSIBLE_AGE_REVERSAL_DAYS:   # < 180
    self.prior_p_substitution = 0.8
elif self.days < 90:                            # НИКОГДА не достигается
    self.prior_p_substitution = 0.4
```
Второй `elif` недостижим — 90 < 180, первый if всегда перехватит `days < 90`. Все окна менее 180 дней получают `prior = 0.8` вместо дифференцированной оценки. Окно 10 дней и 170 дней — одинаковый prior 0.8.

**[CH-04]** В `analyze_timeline` флаг `CHRONO_FLAG_IMPOSSIBLE` генерируется при `dist > deviation_threshold AND prior_p > 0.5`. `deviation_threshold = 1.2` — L2-норма в пространстве метрик. Метрики не нормированы: `gonial_angle ≈ 130.0`, `jaw_width_ratio ≈ 0.65`. Неподготовленный L2-вектор доминируется абсолютными значениями крупных метрик (углы в градусах). Порог 1.2 будет превышен всегда если хоть одна угловая метрика изменилась на 1.2°. Это генерирует ложные IMPOSSIBLE-флаги на каждом шаге временного ряда.

**[CH-05]** `get_ordered_metric_vector` — при отсутствующем ключе подставляет `0.0`:
```python
vector.append(metrics_dict.get(key, 0.0))
```
Для `gonial_angle` (≈130°) замена отсутствующего значения на `0.0` вносит в вектор отклонение ~130 единиц. L2-дистанция к соседнему фото с реальным `gonial_angle=130` будет ≈130 >> порог 1.2. Любое фото с отсутствующей метрикой генерирует IMPOSSIBLE-флаг.

## 4. `quality_gate.py` — Пороги и логика

**[QG-01]** `is_rejected = overall_score < 0.45`. `overall_score = sharpness_score * 0.7 + noise_quality * 0.3`. `sharpness_score = clip(blur_score / 150.0, 0, 1)`. Для отклонения нужен `blur_score < 96.4` (при нулевом шуме). `BLUR_THRESHOLD_DEFAULT = 65.0`. Фото с `blur_score = 70` (выше порога) даст `sharpness_score = 0.467`, `overall_score = 0.327` → **отклоняется** несмотря на то что превышает `BLUR_THRESHOLD`. Внутренняя логика `QualityGate` несогласована с собственной константой `BLUR_THRESHOLD_DEFAULT`. Константа нигде не используется внутри `evaluate()`.

**[QG-02]** `QualityGate.evaluate` принимает `bbox` с ключом `h` (высота bbox лица). В `cascade.py` и `run.py` вызов: `self.quality_gate.evaluate(photo_a)` — без `bbox`. Проверка `if face_h < 150` **никогда не выполняется**. Маленькие лица (< 150px) не отфильтровываются.

**[QG-03]** `_estimate_noise` через `cv2.medianBlur(gray, 3)` — медианный фильтр 3×3. Оценка шума как MAE от медианного. Это стандартный подход для AWGN, но сжатые JPEG дают структурированный блок-шум (8×8 блоки), который медиана 3×3 не подавляет. JPEG-артефактные фото могут получить заниженную `noise_score` → высокую `noise_quality` → пройти gate.

**[QG-04]** `noise_quality = clip(1.0 - noise_score / 25.0, 0, 1)`. При `noise_score = 2.5` (= `NOISE_THRESHOLD_DEFAULT`) → `noise_quality = 0.9`. Порог 2.5 соответствует отличному качеству по этой формуле, а не границе приемлемости. Константа названа `THRESHOLD` но используется как "нормальный уровень".

## 5. `reconstruction.py` — Детали реализации

**[RC-01]** `normals_camera = (self._model.compute_norm(base_shape) @ rotation)` — нормали вычисляются от `base_shape` (без expression deformation), а не от `transformed_shape`. Это означает что нормали соответствуют нейтральному выражению лица, даже если человек улыбается. При открытом рте нормали нижней части лица неверны — маска видимости строится некорректно для dynamic expressions.

**[RC-02]** `trans_params=None if trans_params is None else np.asarray(trans_params)` — `trans_params` из detector возвращается как тензор. `np.asarray(tensor)` не отсоединяет его от графа вычислений если `requires_grad=True`. Потенциальный memory leak при батч-обработке (gradient накапливается).

**[RC-03]** `landmarks_106=None if "ldm106" not in result else np.asarray(result["ldm106"])[0]` — если 3DDFA-V3 не находит лицо, `result` может не содержать `"ldm106"`, и `landmarks_106 = None`. Но в `compare.py` IPD вычисляется через `orbit_L/R` из меша, а не через landmarks. Недоступность landmarks не влияет на comparison — это корректно, но dead code в `recon_a_landmarks_available`.

## 1. Глобальный разрыв: `BUCKET_METRIC_KEYS` vs `result.json`

**[BM-01]** Это **самый системный баг всего проекта**. `BUCKET_METRIC_KEYS` во всех 10 bucket-ах использует ключи вида `texture_lbp_complexity`, `texture_pore_density`, `texture_wrinkle_nasolabial`, `texture_global_smoothness`, `texture_silicone_prob`, `texture_specular_gloss`. **[BM-02]** `texture_lbp_complexity` — такого поля нет нигде в pipeline. **[BM-03]** `texture_pore_density` в `BUCKET_METRIC_KEYS` — тоже несуществующий ключ. В `result.json` поле называется `metrics.texture.nose_pore_density`. **[BM-04]** `texture_silicone_prob` в `BUCKET_METRIC_KEYS` — этот ключ сохраняется в `result.json` как `metrics.texture_forensics.silicone_probability` (через `_compute_texture_h1_evidence`), а не как плоский `texture_silicone_prob`. ## 2. `persona.py` — Кластеризация

**[PS-01]** Алгоритм кластеризации — **Greedy Single-Linkage** без полной матрицы расстояний:
```python
for i in range(len(valid_records)):
    if assigned[i]: continue
    for j in range(i+1, len(valid_records)):
        dist = np.linalg.norm(norm_data[i] - norm_data[j])
        if dist < 0.2: assigned[j] = True
```
При `A ↔ B` (dist=0.15), `B ↔ C` (dist=0.18), `A ↔ C` (dist=0.35) — алгоритм объединит A, B, C в один кластер несмотря на то что A и C несхожи. Истинный single-linkage должен итеративно обновлять расстояния. Это даёт ложные "персоны".

**[PS-02]** Нормализация:
```python
denom = maxs - mins
denom[denom == 0] = 1.0
```
При одной уникальной метрике (все фото одного ракурса имеют `gonial_angle ≈ 130.0` с разбросом 0.5°): `max - min = 0.5` → нормировка корректна. Но если **все** записи имеют одинаковое значение (`gonial_angle = 130.0` везде из-за fallback) → `denom = 0` → заменяется на 1.0 → после нормировки все значения = 0.0 → все расстояния = 0 → **все фото объединяются в один кластер**. При массовом fallback-значении `gonial_angle = 0.0` (баг SC-01 итерации 3) это происходит гарантированно.

**[PS-03]** `"avg_profile"` вычисляется по всем `keys` из `BUCKET_METRIC_KEYS` с `metrics.get(k, 0.0)`. Несуществующие ключи (BM-01) всегда дают 0.0 → avg_profile содержит нулевые текстурные метрики. Профиль персоны структурно неверен.

## 3. `longitudinal.py` — Возрастная модель

**[LG-01]** `LongitudinalAnalyzer.add_photo` при отсутствии года:
```python
if year is None:
    raise ValueError(f"Cannot determine year for photo {photo_id}")
```
Это жёсткий `raise` в публичном методе. Любое фото без извлекаемого года — включая фото с нестандартным именем — прерывает весь longitudinal анализ. `batch_processor.py` вызывает `cascade.analyze_single()`, а не `LongitudinalAnalyzer` напрямую — longitudinal вообще не интегрирован в основной пайплайн.

## Критические (ломают результат)

**[N-01] `specular_gloss` — инвертированная логика порога**
```python
bright_threshold = min(float(p97), 230.0)  # не ниже 230
```
Комментарий говорит "не ниже 230", но `min()` устанавливает **верхнюю** границу 230, а не нижнюю. Если `p97 = 240` → `threshold = 230` (корректно). Если `p97 = 180` (тёмное фото) → `threshold = 180` → `specular_gloss` будет завышен: яркие пиксели при 180 включают всю нормально освещённую кожу. Нужен `max(float(p97), 230.0)` или исправление комментария и логики. 

**[N-02] `quality_index` — Pitch в градусах вместо радиан**
```python
pose_factor = np.cos(np.radians(yaw_deg)) * np.cos(np.radians(pitch_deg))
```
Выглядит корректно, **но** `pitch_deg` передаётся из `run.py` как `res.angles_deg[0]`. При pitch=0° cos(0)=1.0 — нет проблем. Но `yaw_deg` в `texture.py` поступает как абсолютный угол (всегда ≥0°), тогда как реальный yaw может быть отрицательным. `cos(-45°) = cos(45°)` — симметрично, проблемы нет. Однако `pose_factor` при `pitch=30°, yaw=30°` = `0.866 × 0.866 = 0.75`, что занижает `quality_index` на 25% для нормального портретного снимка с лёгким наклоном головы. 

**[N-03] `analyze_texture_synthetic_prob` — дублирует `quality_sharpness_score` без связи**
Функция `analyze_texture_synthetic_prob` (строки ~310–330) вычисляет `Laplacian.var()` и возвращает `synthetic_prob` в диапазоне 0–1 — но это **та же метрика** что и `quality_sharpness_score = cv2.Laplacian(gray, cv2.CV_64F).var()`. При этом функция — отдельный `def` вне класса, вызывается в `run.py` через `_texture_synth_prob = analyze_texture_synthetic_prob(crop_path)`. Результат сохраняется в `result.json` как `synthetic_prob`. В `BayesianMultiHypothesisEngine.synthesize()` этот `synthetic_prob` **не передаётся** — передаётся `texture_silicone_prob` из `HDUVTextureGenerator`. Дублирующий сигнал висит в JSON и нигде не используется в Байесовском движке.

**[N-04] `gpa_unit_scale` — масштаб из shared_idx, центроид из full mesh**
```python
_, scale_a_gpa, centroid_a = gpa_unit_scale(verts_a_canon[shared_idx])
# ...
points_a_unit = (verts_a_canon - centroid_a) / (scale_a + 1e-8)  # full mesh
```
`centroid_a` вычисляется из `shared_idx` (подмножество вершин), но затем вычитается из **полного меша** `verts_a_canon`. Если `shared_idx` не симметричен (профильный снимок: видна одна сторона), центроид смещён в сторону видимых вершин. Нормировка полного меша от смещённого центроида вносит ложное трансляционное смещение в `points_a_unit`. 

**[N-05] `rigid_umeyama` — `scale` формула при `allow_scale=True` неверна**
```python
scale = float((s[0] + s[1] + s[2] * np.sign(d)) / var_source)
```
Стандартная формула Umeyama: `scale = (1/var_source) * sum(s_i * D_ii)` где `D = diag(1,1,sign(det))`. Это `(s[0] + s[1] + s[2]*sign(d)) / var_source` — формально корректно. Но `var_source = np.sum(weights * sum(centered_source**2, axis=1))` — сумма взвешенных квадратных норм без деления на `weight_sum`. Стандартный Umeyama использует **нормированную** дисперсию: `var_source / weight_sum`. Без нормировки scale систематически занижен при неравномерных весах.

**[N-06] `_roi_by_fraction` на изображении 224×224 вместо оригинала**
`wrinkle_forehead`, `nasolabial_depth`, `crow_feet_score`, `nose_pore_density` — все вычисляются через `_roi_by_fraction(gray, ...)` где `gray` — это ресайзнутый кроп лица. Если `face_crop_path` указывает на файл 224×224 (сохранённый 3DDFA-input), то: `roi_forehead = gray[0:67, 34:190]` — 67px высота лба. При оригинальном разрешении 1080p кроп лица может быть 400×400 px. `roi_forehead` в 224px кропе вмещает ~30 пикселей морщины vs 120 в 400px. `laplace(roi).var()` для форхед морщин при 224px даёт в 16× меньшую дисперсию — метрика возраста систематически занижена.

**[N-07] `GLCM` — `graycomatrix` вызывается с `levels=256` на uint8 изображении**
```python
# Основной GLCM в texture.py (строки ~115-125):
glcm = graycomatrix(gray_uint8, distances=[1,2,4], angles=[0,np.pi/4,np.pi/2,3*np.pi/4], levels=256, ...)
```
По документации scikit-image `graycomatrix` с `levels=256` создаёт матрицу 256×256×n_dist×n_angles. Для 3 расстояний и 4 углов: 256×256×3×4 = **786 432 ячейки**. При вызове на каждом кропе в каждом фото (batch из 100 фото) это ~78M float64 элементов (~625 MB RAM) на каждую обработку. Рекомендованный `levels=64` снизит до 9.7 MB. UV-GLCM уже использует `levels=32` — inconsistency.

**[N-08] `euler_to_rotation_matrix` — ZYX vs ZYX convention mismatch**
```python
# В alignment.py:
# R = Rz(roll) @ Ry(yaw) @ Rx(pitch)  — ZYX
```
Но 3DDFA-V3 документация (`recon.py`, строка 47): `R = Rx(pitch) @ Ry(yaw) @ Rz(roll)` — **XYZ convention**. `euler_to_rotation_matrix` используется в `canonicalize_vertices_for_bucket` для поворота к нейтральной позе. При несовпадении convention канонизация вносит неверный поворот: для yaw=45° (профиль) итоговый R будет отличаться от правильного примерно на 3–8° в зависимости от pitch и roll.

**[N-09] `LBP` — `method="uniform"` с `P=8, R=1` даёт 10 паттернов, но `hist` с `bins=256`**
```python
lbp = local_binary_pattern(gray_uint8, P=8, R=1, method="uniform")
hist, _ = np.histogram(lbp, bins=256, range=(0, 256), density=True)
metrics.lbp_uniformity = float(np.sum(hist**2))
```
При `method="uniform"` LBP возвращает значения 0–9 (10 паттернов для P=8). Гистограмма с `bins=256` и `range=(0,256)` создаёт 256 бинов, из которых **заполнены только первые 10**. `lbp_uniformity = sum(hist²)` при 246 пустых бинах всегда около `sum(10_values²)` — фактически нормированная "Херфиндаль-концентрация" только первых 10 бинов. Правильно: `bins=10, range=(0,10)`. UV-LBP (`bins=10`) сделан корректно — ещё одна internal inconsistency.

**[N-10] `score_aligned_pair` — `fit_best_plane` вычисляется из `points_b`, а не aligned shared points**
```python
_centroid_b, plane_normal = fit_best_plane(points_b)  # full points_b
```
`points_b` — все вершины target после unit-scaling. `plane_normal` затем возвращается вызывающему коду для записи в JSON (`"face_plane_normal"`). Но `score_aligned_pair` уже вызывается с `alignment.source_aligned` и `points_b` — оба из `shared_idx` подмножества. Если `points_b` здесь = `verts_b_unit[shared_idx]`, то это partial mesh и нормаль к нему нестабильна при профильных ракурсах (visible set = 30–40% меша).

## Важные (систематически искажают анализ)

**[N-11] `apply_expression_exclusion_mask` — маска не применяется**
```python
return base_mask, excluded_flags
```
Функция возвращает `base_mask` **без изменений**, независимо от флагов `jaw_excluded` и `smile_excluded`. Флаги вычислены корректно, но маска не модифицируется. Вызывающий код должен сам применять флаги — в `zones.py` строки ~89–95 это делается через `EXCLUDED_ZONES`. Но `EAR_INDICES` и expression-exclusion работают на разных уровнях (вершины vs именованные зоны) — часть экспрессионных зон может не исключаться через `EXCLUDED_ZONES`.

**[N-12] `extract_macro_bone_metrics` — `cheekbone_R` аппроксимируется через `brow_ridge_R`**
```python
cheek_R = get_zone_centroid('cheekbone_R')
if np.allclose(cheek_R, 0):
    brow_R = get_zone_centroid('brow_ridge_R')
    # Symmetric approximation: mirror cheek_L across midline
```
`MACRO_BONE_INDICES` не содержит `cheekbone_R` — значит fallback активен всегда. Скуловая ширина `zygomatic_breadth` вычисляется от cheekL до зеркального cheekL — это билатерально симметричная оценка, которая не учитывает асимметрию лица. `interorbital_ratio` нормируется на ложно симметричную `zygomatic_breadth`. 

**[N-13] `nasolabial_depth` — Canny без сглаживания на сжатом JPEG**
```python
metrics.nasolabial_depth = float(np.mean(canny(roi_nasolabial).astype(np.float32)))
```
`canny()` из scikit-image с дефолтными параметрами (`sigma=1.0`). На JPEG-сжатом изображении `sigma=1.0` недостаточно для подавления блок-артефактов 8×8 сжатия в зоне носогубной складки. Для JPEG нужен `sigma ≥ 2.0`. Результат: плотность Canny-рёбер на JPEG сильно коррелирует с уровнем сжатия (качеством JPEG), а не с глубиной морщины.

**[N-14] `spot_density` — порог 1.5σ от среднего включает тени**
```python
metrics.spot_density = float(np.sum(masked_pixels < (mean_val - 1.5 * std_val)) / masked_pixels.size)
```
"Пятна" определяются как пиксели темнее `mean - 1.5σ`. При нормальном распределении яркости кожи `~6.7%` пикселей всегда попадут в этот диапазон даже на идеально чистой коже. Тени от носа, глубоких морщин и складок также дадут тёмные пиксели. Метрика не отличает пигментацию от теней — `spot_density` систематически выше для фото с боковым освещением.

**[N-15] `uv_silicone_flatness = 1 / (1 + glcm_contrast_uv)` — инвертированная семантика**
`glcm_contrast` высокий → много текстурных переходов → натуральная кожа. `uv_silicone_flatness = 1/(1+high_contrast)` → **маленькое значение** для натуральной кожи, **большое** для силикона. Но в `contract.py` `silicone_prob > 0.85 → impossible`. Значение `uv_silicone_flatness = 0.9` (большое) = силикон. Однако в `TextureMetrics` нет прямого маппинга `uv_silicone_flatness → silicone_probability`. Нигде в коде нет `if uv_silicone_flatness > X: silicone_probability = ...`. Метрика вычисляется, сохраняется, но не конвертируется в `silicone_probability`.

**[N-16] `verdict.py` — `CHRONO_FLAG_IMPOSSIBLE` в likelihoods повышает P(H1) и P(H2) одновременно**
```python
# Если IMPOSSIBLE флаг присутствует:
l_chrono = {"H0": 0.01, "H1": 2.5, "H2": 2.5}
```
P(H1)=identity_swap и P(H2)=different_person усиливаются одинаково при IMPOSSIBLE флаге. Но флаг CH-04 итерации 4 генерирует IMPOSSIBLE ложно (ненормированный L2). Это означает каждая пара фото получает Байесовский сдвиг к H1/H2 вне зависимости от геометрии.

**[N-17] `compare.py` строки ~120–125: `pose_delta` через `np.linalg.norm(rot_vec)`**
```python
rot_vec = rotation_matrix_to_axis_angle(alignment.rotation)
pose_delta = float(np.linalg.norm(rot_vec) * 180.0 / np.pi)
```
`rotation_matrix_to_axis_angle` — не видна в импортах `alignment.py`. Возможно использует `cv2.Rodrigues()`. `np.linalg.norm(rot_vec)` = угол вращения в радианах. Умножение `* 180/π` = в градусах. Но `_compute_linear_snr(signal_error, noise_baseline)` принимает `pose_delta` как модификатор шума. Если `pose_delta = 35°` (разные ракурсы между фото), `noise_baseline` увеличивается. Но в `calibration.py` `pair_noise` уже нормализован на позу — двойная коррекция шума.

**[N-18] `scoring.py::_get_face_scale_from_points` — percentile  возвращает [max_approx, min_approx]**[1]
```python
q_x = np.percentile(points[:, 0], [95, 5])
x_extent = q_x[0] - q_x[1]  # 95th - 5th
```
`q_x[0]` = 95-й перцентиль, `q_x[1]` = 5-й перцентиль. При профильном снимке все видимые вершины сжаты по X: 95th-5th по X мал. По Y: 95th-5th нормален. `scale = max(small_x, normal_y * 0.7)` → scale может определяться Y-extent с коэффициентом 0.7. Нормировка ошибки по занижённому scale при профиле завышает `distances_normalized` → ложное увеличение primary_error для профильных пар.

**[N-19] `compare.py` — `landmarks_106` индексы глаз не документированы**
```python
le_a = lm_a[38]  # Left eye inner canthus
re_a = lm_a[88]  # Right eye inner canthus
```
В 3DDFA-V3 `landmarks_106`: точки 38 и 88 — по документации это **внешний** кантус левого глаза и **внутренний** кантус правого. IPD должен меряться между **внутренними** кантусами (более стабильная анатомическая точка). Если оба взяты как внешний+внутренний, IPD не является стандартным межорбитальным расстоянием. 

**[N-20] `bounded_score_from_error` — формула из `utils.py` не верифицирована**
```python
def bounded_score_from_error(error: float) -> float:
    return float(np.clip(1.0 - error / BOUNDED_SCORE_MAX_ERROR, 0.0, 1.0))
```
`BOUNDED_SCORE_MAX_ERROR` из `constants.py`. При `BOUNDED_SCORE_MAX_ERROR = 0.15` и `error = 0.05` → `score = 0.667`. Линейная функция означает что score=0 при error≥0.15. Но после нормировки на `face_scale` (~0.4 для GPA-единичного меша) `error = 0.05` уже умеренное расхождение. Нет калибровки этого порога на реальных парах.

## Требуют исправления

**[N-21] `canonicalize_vertices_for_bucket` — yaw sign зависит от `_CANONICAL_YAW_BY_VIEW_GROUP`**
Словарь: `left_threequarter_light: -22.5`. Поворот к нейтральной позе = `-(-22.5) = +22.5°`. Но если исходный yaw лица = `-20°` (лёгкий левый поворот), то после канонизации face получит `+2.5°` остаточного разворота вправо. При сравнении с правым фото (`right_threequarter_light: +22.5°`, после канонизации `-2.5°`) — оба фото имеют противоположные остатки ±2.5°. Алайнер компенсирует их, но это вносит ложное rotation в `alignment.rotation` которое используется для `pose_delta`.

**[N-22] `texture.py` — `_apply_mask` не проверяет форму маски**
```python
def _apply_mask(arr, mask):
    return arr[mask > 0].flatten()
```
Если `arr.shape = (224, 224)` и `mask.shape = (224, 224, 3)` (BGR маска) — `mask > 0` даёт `(224,224,3)` boolean, и `arr[mask>0]` вызовет `IndexError` или неверную выборку. В коде нет `mask = mask[:,:,0]` перед применением.

**[N-23] `ForensicBatchProcessor.process_directory` — нет создания face_crop**
`cascade.analyze_single(img_path)` передаёт сырой путь к изображению. `texture_analyzer.analyze_image(face_crop_path)` ожидает **кроп лица**. `analyze_single` создаёт кроп внутри (через 3DDFA детектор). Но `uv_path` и `uv_mask_path` в batch processor **не передаются** в `analyze_single`. UV-текстурный анализ (группа 3 в `texture.py`) никогда не выполняется через batch processor — только через `run.py`.

**[N-24] `core/utils.py::parse_date_from_name` — паттерн `(\d{4})(\d{2})(\d{2})` опасен**
Паттерн без разделителей совпадает с любой 8-значной последовательностью включая хэши в именах файлов (`a1b2c3d4` — не совпадает, но `20121234` совпадёт как год=2012, месяц=12, день=34). День=34 не валидируется. `date(2012, 12, 34)` выбросит `ValueError`. Нет try/except вокруг `date(year, month, day)`.

**[N-25] `recommendations.py` — `CalibrationSummary.bucket_coverage` может быть `None`**
```python
calibration_coverage = calibration_summary.get("bucket_coverage", {})
cal_count = int(calibration_coverage.get(bucket, 0))
```
Если `calibration_summary` содержит `bucket_coverage: null` (JSON null → Python None), `.get("bucket_coverage", {})` вернёт `None`. Затем `None.get(bucket, 0)` → `AttributeError`. Нет проверки `if isinstance(calibration_coverage, dict)`.

**[N-26] `verdict.py::synthesize` — CHRONO_FLAG_RETURN повышает P(H0)**
```python
if CHRONO_FLAG_RETURN in flag_types:
    l_chrono["H0"] *= 1.4
```
Флаг RETURN (возврат к baseline после аномалии) интерпретируется как доказательство идентичности того же человека. Но RTR-детектор (CH-02 итерации 4) имеет баги в построении `ref_avg`. Ложный флаг RETURN систематически завышает P(H0) → занижает чувствительность к реальным аномалиям.

**[N-27] `scoring.py::align_and_score` — `alignment_weights` может быть None при наличии `weights`**
```python
alignment = rigid_umeyama(points_a, points_b, weights=alignment_weights, allow_scale=False)
```
`alignment_weights` — отдельный параметр. В `compare.py` вызов: `align_and_score(..., weights=weights, alignment_weights=None)`. При `alignment_weights=None` в `rigid_umeyama` используются равные веса для alignment, а `weights` (неравномерные по видимости) только для scoring. Это допустимо по дизайну, но нет комментария — неочевидно. При совпадении `weights == alignment_weights` (текущий вызов всегда передаёт `None`) alignment использует visibility-взвешивание только через `weights` в scoring. Alignment оптимизирован по равным весам → может быть субоптимален.

**[N-28] `texture.py::quality_index` — `q_index_sharp` при `sharpness_score=0`**
```python
q_index_sharp = 0.87 * (0.0 / 400.0) ** 2  # = 0.0
quality_index = 0.0 * pose_factor = 0.0
```
Совершенно размытое изображение получает `quality_index=0.0`. Но `QualityGate.evaluate` использует отдельный `overall_score` — нет связи между `quality_index` из texture.py и gate score. Фото с `quality_index=0.0` может пройти QualityGate если `overall_score >= 0.45`.

**[N-29] `compare.py` — `shared_vertex_indices` без порога минимального количества**
```python
shared = np.intersect1d(vis_a.visible_indices, vis_b.visible_indices)
```
При профиль-профиль сравнении (оба лица повёрнуты в одну сторону) shared set мал. Нет минимального порога: если `len(shared) < 50` — Umeyama на 50 точках ещё приемлем, но при `len(shared) < 10` результат ненадёжен. В коде только `if len(shared_idx) < 3: return early_return` (аналог `ALIGNMENT_MIN_RANK=3`, баг CN-02).

**[N-30] `batch_processor.py` — `cascade.analyze_single` vs `cascade.analyze_pair`**
`process_directory` вызывает `analyze_single(img_path)` для каждого фото по отдельности. `analyze_single` не выполняет попарное сравнение — нет вызова `analyze_pair`. Весь `PairComparisonEngine`, `CalibrationAnalyzer`, `BayesianMultiHypothesisEngine` никогда не запускаются через batch processor. `forensic_bundle.json` содержит только одиночные паспорта без форензического вердикта "одно лицо / разные лица". Основная задача проекта (сравнение лиц) не выполняется в batch-режиме.

## Приоритет P0

Сначала нужно остановить все источники систематически неверных выводов. Пока эти узлы не исправлены, любые локальные улучшения интерфейса, рекомендаций, longitudinal и persona не имеют смысла, потому что базовые метрики уже могут быть ложными.

**1. Убрать расхождение схем данных между `run.py` и `backend/core/utils.py`.**  
Сейчас один из самых тяжёлых дефектов — `BUCKET_METRIC_KEYS` ссылается на несуществующие ключи: `texture_lbp_complexity`, `texture_pore_density`, `texture_wrinkle_nasolabial`, `texture_global_smoothness`, `texture_silicone_prob` и часть других. В `result.json` реальные текстурные метрики лежат во вложенной структуре `metrics.texture.*`, а часть сигналов живёт в `metrics.texture_forensics.*`, а не в плоских полях. Из-за этого `persona.py`, `longitudinal.py`, часть summary-логики и привязка bucket-метрик получают нули или `None` без явной ошибки. Исправление должно быть не точечным, а архитектурным: ввести единую каноническую схему метрик, например `MetricRegistry`, где для каждой метрики задано имя, путь в JSON, допустимый диапазон, bucket availability и тип нормализации. После этого полностью переписать `BUCKET_METRIC_KEYS`, `FORENSIC_RADAR_AXES`, `persona.py`, `longitudinal.py`, `service.py` и все места чтения `metrics.get(k, 0.0)`.

**2. Убрать все дубли классификации ракурса и оставить один источник истины.**  
Сейчас в проекте минимум шесть несовпадающих реализаций `classify_pose_bucket`: в `run.py`, `bucket_from_filename`, `reconstruction.py`, `core/analysis.py`, `core/utils.py` и частично в `detect_pose.py`. Пороги отличаются: frontal где-то до 10°, где-то 12°, где-то 15°; profile где-то с 65°, где-то с 70°. Это критично, потому что bucket определяет калибровку, разрешённые метрики, интерпретацию longitudinal и pair-comparison. Нужно вынести единственную функцию `classify_pose_bucket(yaw_deg)` в один модуль, например `pipeline/types.py` или `core/constants.py`, зафиксировать пороги на основании ТЗ и использовать её везде без копий. Дополнительно нужно ввести regression-тесты на граничные углы: 10, 12, 15, 25, 45, 65, 70 градусов.

**3. Исправить несогласованность координатных пространств в UV-пайплайне.**  
Сейчас в генерацию UV передаются канонизированные 3D-вершины и неканонизированные 2D-проекции. Это одна из самых опасных ошибок, потому что она визуально не всегда очевидна, но искажает barycentric bake, visibility и текстурные признаки. Нужно выбрать один строго согласованный режим: либо UV строится целиком в исходном пространстве реконструкции, либо сначала канонизируются и 3D, и 2D данные согласованным преобразованием. Параллельно надо убрать двойную/конкурирующую логику масок: маска от UV-generator, маска из normals и последующая эрозия сейчас конфликтуют между собой.

**4. Переписать текстурный слой так, чтобы метрики реально соответствовали своим названиям.**  
Сейчас там много дефектов: `LBP` считается с `method="uniform"`, но гистограмма строится на 256 bins вместо 10; `specular_gloss` использует перевёрнутую логику `min/max`; `spot_density` ловит тени и освещение вместо реальных пятен; `nasolabial_depth` на JPEG зависит от компрессии; `uv_silicone_flatness` считается, но не переводится в `silicone_probability`; `synthetic_prob` вычисляется отдельно и не участвует в итоговом Bayesian verdict. Нужен рефактор в три слоя: базовые текстурные признаки, forensic-признаки, quality-driven gating. Каждая метрика должна иметь чёткое определение, диапазон, unit test на synthetic input и понятную интерпретацию.

**5. Исправить математическую основу геометрического сравнения.**  
Есть несколько дефектов сразу: `reliability_weight` в `score_aligned_pair` фактически не влияет на итоговый score из-за нормировки; `gpa_unit_scale` считает scale/centroid по shared subset, а затем нормирует полный меш; `shared_vertex_indices` не имеет минимального осмысленного порога; `score_aligned_pair` использует нестабильную plane normal для частичного меша; face scale завышает ошибки на профилях. Нужно пересобрать весь тракт `canonicalize -> shared_idx -> unit_scale -> rigid_umeyama -> normalized_error`. Важный принцип: scale, centroid, alignment set и scoring set должны быть явно разделены и документированы. Для partial/profile views нужно отдельное правило нормализации, а не тот же frontal-scale.

## Приоритет P1

Это следующий слой: код уже должен считать не “совсем неправильно”, а устойчиво и воспроизводимо. Здесь основная задача — убрать ложные срабатывания и немые ошибки.

**6. Пересобрать калибровку как отдельный строгий контракт.**  
Сейчас `run.py` дублирует логику стадий, `main` и `calibration` делают почти одно и то же, а fallback baseline `0.015` используется как универсальный шум для разных метрик. Это недопустимо. Нужно объединить стадии 1 и 2 в единый extraction pipeline, а калибровку оформить как вторичный шаг поверх уже извлечённых метрик. Для каждой метрики должен быть свой baseline noise model, а не общая константа. Также нужно централизовать `find_calibration_match`, `CalibrationProtocol`, `pair_noise`, `global_noise_baseline` и исключения по smile/jaw в один модуль.

**7. Исправить Bayesian verdict и связку evidence channels.**  
Сейчас chronology flags могут поднимать H1 и H2 одновременно, `RETURN` искусственно завышает H0, `acceptable` ставится даже при отсутствии данных, а texture channels либо не передаются, либо вычисляются отдельно и не используются. План: сделать явный список входов verdict-движка — geometry evidence, texture synthetic evidence, texture silicone evidence, chronology evidence, data sufficiency. Для каждого канала нужен режим: calibrated, fallback, unavailable. Если канал unavailable, это должно снижать confidence, а не давать “acceptable”. Приоритетно переписать `contract.py`, `verdict.py` и итоговую сборку `global_status`.

**8. Развести extraction и analysis на уровне архитектуры.**  
Сейчас `run.py` — монолит. Нужно вынести независимые стадии: detection/crop, reconstruction, visibility/canonicalization, geometry extraction, texture extraction, calibration matching, pair analysis, reporting. На выходе каждой стадии должен быть свой артефакт с фиксированной схемой: например `photo_manifest.json`, `reconstruction.json`, `geometry.json`, `texture.json`, `pair_result.json`. Это решит проблему повторного расчёта, облегчит отладку и позволит безопасно переиспользовать данные для будущего интерфейса.

**9. Убрать “молчаливые fallback-значения”.**  
Сейчас слишком много мест, где отсутствие данных превращается в `0.0`, `0.5`, `0.7` или `acceptable`. Именно это и скрывает реальные ошибки. Политика должна быть иной: `None` остаётся `None`, а дальше слой-валидатор обязан пометить `metric_status = unavailable` и объяснить почему. Это касается geometry, texture, calibration, summary, persona, recommendations и global verdict.

**10. Починить batch-обработку как главный рабочий режим.**  
Сейчас `batch_processor.py` не вызывает полноценный pair-analysis, не использует UV path, не учитывает chronology, содержит импорт несуществующего `cleanup_artifacts`, а `EXCLUDED_FROM_ANALYSIS` не работает из-за несовпадающих ID. Нужно переписать batch processor как orchestrator: сначала extraction для всех фото, затем grouping by bucket, потом pair graph / chronology, потом verdict aggregation. Только тогда проект будет соответствовать ТЗ “сначала извлечь все данные и завершить анализ, потом подгружать в интерфейс”.

## Приоритет P2

После исправления критики и основной математики можно стабилизировать продвинутые аналитические слои.

**11. Переписать `persona.py`.**  
Текущая кластеризация greedy и плохо переносит нулевые/отсутствующие метрики. Её надо заменить на алгоритм, который умеет работать с неполными признаками: хотя бы agglomerative clustering с distance mask или DBSCAN по валидным feature-subspaces. Но это имеет смысл только после исправления схемы метрик. Также profile persona должен хранить не среднее по всем ключам с нулевыми fallback, а агрегировать только реально доступные признаки и считать coverage.

**12. Переписать `longitudinal.py`.**  
Сейчас temporal model жёстко падает без даты, slope/variability не согласованы с реальными диапазонами признаков, а события внутри одного года неупорядочены. Нужно вводить явную temporal granularity: exact date, year-month, year-only, unknown. Если даты нет — longitudinal не должен падать, а должен помечать photo как unusable_for_time_series. Age trends должны быть заданы только для реально валидных метрик и в их реальных единицах. Также chronology и longitudinal надо развести: chronology — forensic transitions между фото; longitudinal — медленные естественные изменения.

**13. Нормализовать quality layer.**  
Сейчас есть `QualityGate`, `quality_metrics`, `texture.quality_index`, `synthetic_prob`, local sharpness — несколько параллельных оценок качества без общего решения. Нужно сделать единый quality contract: detection quality, crop quality, reconstruction confidence, texture usability, pose usability. Потом использовать этот contract как gating для geometry и texture, а не просто как декоративные поля в JSON.

**14. Проверить все геометрические формулы на анатомическое соответствие ТЗ.**  
Есть рискованные места: центроиды зон вместо landmark-анатомии, `cheekbone_R` через brow approximation, canthal/IPD индексы 106-landmark модели, nasal and mandibular formulas. Тут нужен отдельный слой верификации “метрика → анатомическое определение → используемые вершины → ракурс, при котором допустима”. Без этого даже математически чистый код может считать не ту величину, что заявлена в ТЗ.

## Приоритет P3

Это финальный слой доведения до production-ready состояния.

**15. Ввести жёсткую систему контрактов и тестов.**  
Нужны unit tests на `classify_pose_bucket`, `LBP`, `GLCM`, `specular_gloss`, `uv_silicone_flatness`, `gpa_unit_scale`, `rigid_umeyama`, `bounded_score_from_error`, chronology transition logic, verdict synthesis. Нужны schema tests для всех JSON-артефактов. Нужны regression datasets: frontal/frontal same person, frontal/profile same person, different person same pose, low-quality images, smile/open-mouth cases.

**16. Убрать хардкод путей и сделать воспроизводимую конфигурацию.**  
`sys.path.insert` с абсолютными путями, относительные storage-paths, невалидируемые константы — всё это надо перевести в `config.py` + environment variables. Для каждой версии pipeline нужен config snapshot, который сохраняется рядом с результатом.

**17. Подготовить данные под будущий интерфейс, но не UI.**  
После рефактора нужно определить финальный набор файлов, который интерфейс потом будет читать без дополнительного пересчёта: `photo_index.json`, `pair_matrix.json`, `bucket_summary.json`, `forensic_manifest.json`, `quality_report.json`, `timeline_report.json`, `calibration_report.json`. То есть интерфейс должен быть только слоем визуализации, а не вычислений.

## Итоговая последовательность работ

Сначала я бы делал не “по файлам”, а по потокам данных.

**Шаг 1:** зафиксировать каноническую схему JSON и registry всех метрик.  
**Шаг 2:** заменить все классификаторы позы на одну функцию.  
**Шаг 3:** пересобрать extraction pipeline: reconstruction, geometry, texture, UV — без verdict.  
**Шаг 4:** переписать geometry normalization и pair comparison.  
**Шаг 5:** переписать texture metrics и forensic texture evidence.  
**Шаг 6:** пересобрать calibration и chronology на уже валидных метриках.  
**Шаг 7:** переписать Bayesian verdict и summary/contract.  
**Шаг 8:** стабилизировать batch processing и выходные артефакты.  
**Шаг 9:** только потом возвращаться к persona, longitudinal, recommendations.  
**Шаг 10:** покрыть всё тестами и прогнать на эталонном наборе фото.

Главная мысль: проект сейчас нужно не “подлатать”, а привести к единой вычислительной модели. Если просто чинить отдельные баги без унификации схемы данных, классификации позы и трактов geometry/texture, ошибки будут возвращаться в новых местах.
